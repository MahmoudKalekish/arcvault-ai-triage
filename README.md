# ArcVault AI Intake And Triage Workflow

This repository contains the materials generated by Codex for the Valsoft AI Engineer take-home assignment. These files are documentation and copy-paste helpers only. `n8n` is **not** connected to this folder automatically, so the workflow was built manually in n8n.

## What This Submission Is

This is a minimal, practical solution for the ArcVault intake and triage scenario:

- orchestration: `n8n`
- LLM provider: `Groq`
- LLM integration method: `HTTP Request` node using an OpenAI-compatible chat completions endpoint
- final downloadable artifact: `outputs.json`
- trigger: `Schedule Trigger`

The design is intentionally small enough for a realistic 3 to 5 hour take-home. It avoids unnecessary infrastructure and keeps the business logic easy to explain in an interview.

## Real Workflow To Build In n8n

The final implemented workflow is:

1. `Schedule Trigger`
2. `Code: Load Sample Inputs`
3. `HTTP Request: LLM Classifier`
4. `Code: Parse LLM JSON`
5. `Code: Route And Escalate`
6. `Aggregate`
7. `Code: Build Outputs JSON`
8. `Convert to File`

Important clarifications:

- There is **no** `Split Out` node in the final design.
- `Code: Load Sample Inputs` already returns one item per request.
- There is **no** `Write Binary File` node in the final solution.
- In `n8n` cloud, the final JSON is produced by `Convert to File` and downloaded from the workflow UI.
- Codex did **not** create or sync the n8n workflow for you.

## Why This Meets The Assignment

This workflow satisfies the Valsoft requirements as follows:

- automatic start: `Schedule Trigger`
- raw message ingestion: sample requests emitted by `Code: Load Sample Inputs`
- LLM classification: one Groq call per message
- exact category set enforced in the prompt
- priority and confidence returned in structured form
- extraction of core issue, identifiers, urgency signal, and summary
- deterministic routing to `Engineering`, `Product`, `Billing`, `IT/Security`, or `Human Review / Escalation`
- deterministic escalation outside the model
- final structured output generated as `outputs.json`

## LLM Responsibility Vs Business Rules

The LLM is responsible only for:

- category
- priority
- confidence
- core issue
- identifiers
- urgency signal
- summary

The `Code: Route And Escalate` node is responsible for:

- destination queue mapping
- low-confidence escalation
- outage and broad-impact escalation
- billing discrepancy threshold escalation

That separation is deliberate. It keeps operational decisioning deterministic, transparent, and easy to defend.

## Assumptions

- A scheduled batch run over the five synthetic messages is acceptable as an automatic trigger for the assessment.
- A downloadable JSON artifact is acceptable as the structured output deliverable.
- Groq is used because it is simple, cheap, and OpenAI-compatible, not because the assignment requires a specific provider.

## Known Limitations / Tradeoffs

- The workflow uses synthetic sample inputs from a code node rather than a live inbound source such as email or webhook.
- The final JSON is downloaded from `Convert to File` in the n8n UI instead of being written to disk automatically in n8n cloud.
- The workflow relies on one LLM call per message and does not include a second validation pass.
- The solution is optimized for clarity and completeness in a take-home assignment, not for production-scale operations.

## What I Would Improve In Phase 2

- Add schema validation and retry handling around the LLM step.
- Persist outputs to a more operational destination such as Google Sheets, Airtable, or a database.
- Add notifications for escalated items and lightweight monitoring around failures.

## Common Failure Modes And Fixes

### Malformed JSON

- Cause: the model returns prose, markdown, or extra wrapper text.
- Fix: keep `temperature: 0`, use `response_format.type = json_object`, and use the strict system prompt in [prompts.md](/home/mahmoud-in2/Desktop/valsoft/prompts.md:13).

### Wrong Field Types

- Cause: the prompt is too loose and the model invents values like `"High"` for confidence or an object for identifiers.
- Fix: keep the exact schema constraints and explicitly require `confidence` as an integer, `identifiers` as an array of strings, and `urgency_signal` as one of `low`, `moderate`, or `high`.

### Auth Issues

- Cause: missing or invalid Groq API key.
- Fix: verify the `Authorization` header is `Bearer <GROQ_API_KEY>` and confirm the key works in a simple test request.

### Parse Failures

- Cause: the parse node expects `choices[0].message.content` but the HTTP node output shape differs.
- Fix: inspect one raw HTTP response in n8n, confirm the field path, and update the parser if needed. The expected Groq OpenAI-compatible response path is `choices[0].message.content`.
